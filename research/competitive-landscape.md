# Competitive Analysis: AI Design/App Builders vs. Design Book

**Synthesized from four deep-dive reports:** Lovable, Vercel v0 + bolt.new, Replit Agent (v1→Agent 4), and a cross-product technique survey (Playwright MCP, Chrome DevTools MCP, Stagehand, Onlook, plus published evals: UI2Code^N, VisRefiner, AniMINT, WebArena ablations).

**Grounded against:** `designbook/ARCHITECTURE.md`, `lib/inspect.js` (viewport lab), `lib/book.js` (pages/briefs store), `designbook-mcp/server.js` (locked tool surface), `engines/sdk.js` (Agent SDK engine), `server.js` (HTTP API).

---

## 1. The Landscape

| Product | Architecture bet | Verification instrument | Cheap tier | Recovery mechanism |
|---|---|---|---|---|
| **Lovable** | Single iterative agent (explicitly abandoned multi-agent), persistent Fly.io sandbox per project (4,000+ machines), Vite+HMR | Console/network log tools, build errors over WebSocket, design-system adherence scan with auto-retry | Client-side AST + Tailwind generator (zero-LLM visual edits, free inline text) | Every edit = a commit; bookmarks, branching; docs say "revert, don't keep fixing" |
| **v0 (Vercel)** | Composite model pipeline: RAG + frontier base (Claude) + RFT-trained AutoFix model repairing the stream itself | Real builds; agent browser-testing w/ screenshots (2026); one org-wide KPI: % of generations yielding a working preview | Quick Edit small model; Design Mode (zero-generation visual edits) | Diffable/revertable versions; forking |
| **bolt.new** | Node-in-WASM (WebContainers) in the browser tab; one mega-prompt + streamed XML action protocol, no function calling | Dev-server/terminal error auto-capture, "Attempt fix" | Plan mode (discuss without generating); zero-token version restore | Version history restore (explicitly marketed as free undo) |
| **Replit Agent 3/4** | Fully-managed cloud world; verification = code-driven Playwright in a persistent REPL, run as an isolated subagent | Stripped ARIA DOM + delta logs (since last execution) + read-only SQL; no screenshots in the agent loop | Lite mode; Design Canvas edits that bypass the agent loop entirely | Atomic checkpoints: git commit + Neon DB branch + agent memory, bidirectional rollback |
| **Tooling layer** (Playwright MCP, DevTools MCP, Stagehand, Onlook) | a11y-tree-as-text with stable element refs; CDP-native; DOM↔source instrumentation (data-oid) | Source-mapped console, network lists, pre-digested perf insights ("LCP is render-delayed, not network-bound") | Structured snapshots are 80–90% smaller than DOM; screenshots opt-in | n/a |

Key context numbers that anchor everything below:
- **v0:** 93.87% error-free generations vs **64.71% for raw Claude Sonnet** on identical tasks — the entire delta is the deterministic wrapper, not the model (the base *is* Claude).
- **Replit:** REPL-based facts verification = **~$0.20 median per multi-hundred-step test session** vs ~$0.50 and 30–90s for a 5-field form under screenshot computer-use. 3x faster, 10x cheaper. Subagent isolation extended useful autonomy from ~20 min to 200+ min.
- **WebArena ablations:** removing the a11y tree drops task success 46.9%→36.7%; removing screenshots costs only ~2 points. Structure carries ~95% of success at ~10–20% of the token cost.
- **UI2Code^N:** iterative render-and-refine gains flatten after 3–5 rounds; pairwise comparison ("which is better?") beats absolute scoring for aesthetic judgment, 94% VLM-human agreement.

## 2. Per-Product Mechanisms (technical)

### Lovable
- **Compile-time component IDs.** A custom Vite plugin stamps every JSX element with a stable unique ID. Click any DOM node → exact JSX source. This single mechanism underpins visual edits, select-to-prompt, and precise diffs. *Design Book gets this nearly free: `composePage` assembles from known vault snippets, so provenance can be stamped at compose time without any AST machinery.*
- **Zero-LLM visual edits.** The project syncs into the browser as a Babel/SWC AST; style/text tweaks mutate the AST directly with a client-side Tailwind generator — zero network, zero tokens. The long tail of "nudge this padding" never touches a model.
- **Select-element-as-context.** Cmd-click elements attach as structured references to the chat input. Pointing replaces describing; the model cannot edit the wrong component.
- **Hydration + context discipline.** A small fast model selects the relevant file subset before Sonnet edits; "useful-context" bookkeeping forbids re-reading; line-replace with ellipsis matching instead of full rewrites; mandatory parallel tool batching. Their stated finding: *context engineering matters more than model choice.*
- **Design-system enforcement in three layers:** prompt-level token rules ("never text-white, always semantic tokens"), machine-readable `design-system.json` (tokens + component catalog + constraints), and a post-generation **adherence scan with automatic retry** — violations (raw color literals, off-scale values, inline-style bypasses) are caught and re-prompted before the user ever sees output.
- **Versioning as the escape hatch.** Every AI edit is a commit; official guidance after ~3 failed fixes is revert, not more fixing.
- **Prompt back-testing.** Prompt changes are re-run against a library of historical user queries before deploy. Prompts treated as code with tests.

### Vercel v0
- **Composite pipeline.** RAG + swappable frontier base + `vercel-autofixer-01` (small RFT model) repairing errors *in the output stream* at ~8,130 chars/sec, plus deterministic AST autofixers (<250ms) and dependency completion post-stream.
- **"LLM Suspense" stream rewriting.** Every lucide-react export lives in a vector DB; a hallucinated `import { VercelLogo }` is rewritten via embedding search in <100ms. Hallucinated identifiers — the #1 cause of blank previews — are fixed deterministically at zero token cost.
- **One KPI.** The whole org optimizes a single number: % of generations producing a working preview. Every subsystem aligns to it.
- **Design-system grounding.** Generation pinned to shadcn/Tailwind/Radix; customer systems fed as a shadcn Registry JSON; **cache-stable prompt prefix** engineered to maximize prompt-cache hits; curated read-only directories of "code samples designed for LLM consumption."
- **Tiered edits.** Quick Edit small model for tiny deltas; Design Mode for zero-generation visual edits (every change still diffable/revertable); element-scoped NL edits auto-attach a cropped screenshot of just the selected element.
- **Multi-variant generation.** Classic pick-from-3 flow front-loads taste decisions — but costs v0 3x generation, which is why it was de-emphasized. *Design Book gets variants free from deterministic seeds.*

### bolt.new
- **WebContainers.** Node-in-WASM in the tab; npm install <500ms via CDN-precompressed layers; hot reload in tens of ms. The runtime *is* the eval harness — verify-after-every-write is economically viable because execution is free.
- **Streamed XML action protocol.** No function calling: `<boltArtifact>` with ordered `<boltAction type="file|shell|start">` ops parsed from the stream. Strict ordering encodes environment invariants (deps → files → start, never restart a running server).
- **Full-file emission as a reliability choice.** "ALWAYS provide the FULL updated content" — they never solved patch robustness; diff mode is paywalled and off by default. The cost of reliability was shifted to the user's wallet.
- **Token-economy as product surface:** Plan mode, `/clear`, zero-token version restore, explicit user guidance to narrow scope.
- **The cautionary tale:** "Attempt fix" keeps no memory of prior attempts and assumes the last change was correct — the root of doom loops that have burned users 7–20M tokens on a single bug.

### Replit Agent
- **REPL-based verification (the big one).** The testing agent writes raw Playwright JavaScript in a sandboxed Node VM notebook; variables AND the browser session persist across execution blocks (an order ID captured in step 3 is used in step 12 without re-passing through context). Each execution returns: stripped ARIA-annotated DOM (not screenshots), **client+server logs since the last execution** (delta semantics → cause-and-effect attribution), and read-only DB queries to catch "Potemkin interfaces" — UIs that render but whose client↔server dance is fake.
- **Verifier as isolated subagent.** Main agent sends a high-level assertion plan; subagent loops autonomously; only a summarized pass/fail returns. Memory compressed at every boundary. This is what extended autonomy 10x.
- **Code-mode tool invocation.** Python DSL instead of JSON function calling: ~90% valid invocation across 30+ tools, multiple dependent calls per model turn.
- **Whole-world checkpoints.** Git commit + Neon DB branch (copy-on-write) + agent conversation memory, snapshotted atomically, bidirectionally rollback-able. Safe undo is what makes users tolerate bigger agent swings.
- **Harness-enforced policy.** Reflection forced every 5 steps; rollback-and-restart on stuck; ask-human as a tool call with a 10-minute takeover window then auto-skip. Policies live in the harness, not the prompt — the model can't skip them.
- **State replay for the vendor.** Full agent state in LangSmith traces; failures replayed against proposed harness fixes before shipping.

### The tooling/research layer
- **Playwright MCP / Stagehand:** a11y-tree snapshots with stable refs; all actions target refs, never coordinates; 80–90% payload reduction vs raw DOM; screenshots opt-in fallback. *Refs are what make a loop closed rather than merely observable.*
- **Chrome DevTools MCP:** `list_console_messages` with source-mapped stacks, `list_network_requests`, `performance_analyze_insight` returning *conclusions* instead of 50MB traces — the trace-equivalent of Design Book's layout facts.
- **Onlook:** `data-oid` attributes injected at build time map every DOM node to its source line; visual edits are zero-model.
- **Research findings to bank:** pairwise ranking beats absolute aesthetic scoring (UI2Code^N RVPO); iteration gains flatten after 3–5 rounds; VLMs hit only 64% on animation *purpose* and systematically fail on animations covering <15% of screen; motion-encoded single images (decaying-transparency frame blends) + textual captions beat raw video; verbose unfiltered a11y dumps *hurt* decision quality as much as naive screenshots.

## 3. Cross-Cutting Patterns

Every successful product converged on the same six-part architecture. This is the industry's collective answer to "why does a wrapped agent beat raw Claude":

1. **Close the loop with environment facts, automatically.** Console errors, source-mapped stacks, failed requests, build output, server logs — captured and attached to the agent without the user pasting anything. This is the single biggest gap between products and raw chat, cited in all four reports. The best version (Replit) uses **delta semantics**: facts *since the last action*, so cause attributes to the most recent change.
2. **Deterministic repair before model engagement.** v0's stream rewriting and <250ms AST autofixers; Lovable's adherence scan + auto-retry; bolt's ordered action protocol. The cheapest, most common errors never reach a model turn. This is responsible for v0's 29-point reliability gap over raw Claude.
3. **Addressable observations (stable refs).** Lovable's compile-time IDs, Onlook's data-oid, Playwright MCP's snapshot refs. An observation you can't point back at is a description; an observation with a ref is an action target. Pointing replaces describing, killing both prompt ambiguity and wrong-element edits.
4. **Tiered cost paths.** Zero-token tier (visual edits, version restore, re-theme) → small-model tier (Quick Edit, hydration file-selection) → frontier tier (structural work). Products that lack cheap paths (Replit's opaque effort pricing, bolt's full-file token burn) generate documented user rage ($1,000/week bills, 20M-token doom loops).
5. **Constrained design vocabulary.** v0 and Lovable both pin generation to a token/component system fed to the model as machine-readable JSON (shadcn Registry, design-system.json) with a cache-stable prompt. Constraining degrees of freedom is *why* their output looks designed instead of generic — the model predicts within a coherent system instead of free-styling CSS.
6. **Versioning is feedback architecture, not a convenience.** Every edit a commit; revert documented as the correct exit from fix loops; Replit checkpoints code+data+memory atomically. Recovery beats regeneration.

Plus three second-order patterns:
- **Harness-enforced policy beats prompt-suggested policy** (Replit's forced reflection vs bolt's prompt-only discipline — guess which one doom-loops).
- **Context isolation with summarization at boundaries** (verifier subagents returning compact verdicts; hydration models pre-selecting context).
- **The vendor regression-tests its own harness** (Lovable's prompt back-testing, Replit's trajectory replay, v0's single KPI).

**And one universal blind spot:** every commercial loop is *functional*, not *aesthetic/layout-aware*. Lovable's loop is pixel-blind (layout regressions caught by the human eye); bolt misses browser-console errors entirely (white-screen case); Replit's DOM+log approach "catches broken, not ugly"; neither v0 nor anyone else feeds the model machine-readable layout facts. Design-QA linting (token adherence, spacing scales, alignment) lives in Figma plugins, not live-DOM agent loops. **This is open territory and it is exactly where Design Book already sits.**

## 4. What Design Book Already Does Better (honest assessment)

1. **Facts-not-pixels is the validated, convergent answer — and ours is ahead.** Replit measured 3x faster / 10x cheaper for structured facts over computer-use; WebArena ablations show structure carries ~95% of success at a fraction of the cost. But Replit's facts are a stripped DOM + logs; Stagehand's are an a11y tree. **Nobody ships structured *layout* facts** — overflows, sub-44px tap targets, tiny text, landmark geometry, offscreen content, plus a perf census (forced-reflow cost, style-recalc, long tasks, CLS, blur/animation/DOM-weight census). `lib/inspect.js` is genuinely past the commercial state of the art on the *read* side.
2. **Deterministic-first is a moat, not a feature.** Lovable built a client-side AST engine and a Tailwind generator to get a zero-token tier; v0 trained a custom fix model; Replit added Lite mode after pricing rage. Design Book's Layer 0 (compose, variants, re-theme, coherence, inspection) is zero-token *by construction*. Variants that cost v0 3x generation cost us a seed integer.
3. **Deterministic compose makes everything downstream cheaper.** A checkpoint for Replit is a git commit + a Neon DB branch + serialized agent memory. A checkpoint for Design Book is `{seed, theme tokens, brief, snippet ids}` — a few hundred bytes that perfectly reproduce a page. Their regression testing needs snapshot infrastructure; ours replays exactly. Their DOM↔source mapping needs a Vite plugin; ours is an attribute stamped at compose time.
4. **Dual engines with one tool surface.** No competitor lets the user choose between their existing Claude Code subscription and API credits over an identical MCP tool surface. Lovable/v0/bolt/Replit are all total platform lock-in with metered loops; Design Book's engine-parity invariant means every harness improvement lands for both.
5. **Local-first, plain files, zero-build.** The entire Lovable feedback loop requires 4,000+ cloud sandboxes; bolt requires a tab with SharedArrayBuffer; Replit's powers exist only in their cloud. Design Book's book is a folder a human can read, git, or hand to any agent.
6. **A coherent vault instead of shadcn monoculture.** Lovable's output is "overwhelmingly shadcn-shaped"; v0 is stack-locked. 931 snippets + 55 palettes + a taste/aesthetic token layer is a richer, more distinctive constrained vocabulary — it just isn't yet packaged for LLM consumption (see gaps).

## 5. The Gap List

Ordered by how much the competition's experience depends on each:

1. **No runtime-error channel.** A composed page that throws, 404s its `/vault/*` CSS, or fails a font load is invisible to `ViewportReport`. Every product has this; it's the #1 convergent feature. (Already acknowledged on the roadmap.)
2. **Facts aren't addressable.** `ViewportReport` cites CSS selectors (`sel`), not stable refs. The agent can *see* "overflow at .hero > div:nth-child(2)" but has no handle to patch exactly that subtree. No `book_patch` — the only write is whole-page `book_save_page`, which forces full-page regeneration for one-element fixes (bolt's token-burn mistake).
3. **No validation gate on writes.** `book_save_page` accepts whatever the model produced. No adherence scan (raw hex where a palette token exists, off-scale values, hand-rolled markup duplicating a vault snippet), no auto-retry, no deterministic autofix pass. Coherence scoring exists but isn't enforced as a gate.
4. **No loop-bound or attempt memory in the harness.** Nothing stops engine fix-loops; the doom-loop guard exists in no layer. (Uniquely, our structured facts make loop detection *deterministic* — a fact fingerprint either survives a fix or it doesn't — which no competitor can do with prose logs.)
5. **Revisions exist but aren't a recovery system.** `revisions/<n>.html` is written, but there's no `book_restore_revision` tool, no fact-report stored per revision, no "revert beats regenerate" doctrine in tool descriptions.
6. **The vault isn't packaged for the model.** No registry.json / machine-readable catalog, no cache-stable prompt prefix, no curated exemplars. The model can `/api/search` but has no compact system-level picture of the vocabulary it must compose within — the mechanism behind v0's and Lovable's "looks designed" quality.
7. **Inspection is stateless.** Each `book_inspect` is a cold render. No persistent session, no interaction — hover states, modals, accordions, sticky-header behavior are unverifiable. This is the design-work analog of Replit's "Potemkin interface" class.
8. **No design-lint facts.** Contrast ratios, spacing-scale adherence, sibling bounding-box overlap, alignment clustering, dead controls (href="#", handler-less buttons) — none in the probe. Open territory no commercial product occupies.
9. **No zero-token per-element edit tier.** Instant re-theme is whole-page; text/spacing/palette-slot tweaks per element still need a model turn.
10. **No context-discipline patterns in the SDK engine.** Full ViewportReports across 3 viewports × N iterations flow into the main trajectory; no verifier isolation, no plan/read-only mode, no `book.md` knowledge file.
11. **No regression harness.** Briefs history isn't replayed when compose logic or prompts change; no MCP-call trajectory log.
12. **Animation/motion blind** (already on roadmap; the research now gives the spec: virtual-time frame capture, motion-overlay strips, structured animation census via `document.getAnimations()` — never raw video to a VLM).

## 6. Recommendations Summary

Twelve deduplicated, ranked items follow in the structured list. The dependency spine: **refs (1) → error facts (2) → save-gate (3) → checkpoints (4) → loop guard (5)** form the MVP's closed loop; **registry (7) + design-lint (8) + interact (9)** make it the best verification instrument in the category; **Design Mode (10) + context discipline (11) + back-testing (12)** compound the moat. Deliberately excluded as standalone recs (covered in-report): screenshot-evidence-on-completion (fold into checkpoints), visual-drift scoring and pairwise VLM judging (escalation-ladder roadmap, after design-lint), animation facts (already on the ARCHITECTURE.md roadmap; the AniMINT findings above are the spec).

## Ranked recommendations

1. **Stamp stable provenance refs at compose time; cite them in facts; add book_patch(slug, ref, ...)** (medium effort, high impact, phase: now)
   The keystone — Lovable needs a Vite plugin and Onlook a build-time injector for this; compose_page gets it free. In the composition path (lib/vault.js bridge to composePage), stamp every section/snippet root with data-db-ref="<section-ordinal>.<snippet-id>.<seed>". Change lib/inspect.js so every ViewportReport finding (overflowers, smallTaps, tinyText, offscreen, landmarks) carries the nearest ref alongside the CSS selector. Then add MCP tools: book_scope(slug, ref) returning just that snippet's source + its token bindings as edit context, and book_patch({slug, ref, html|cssVars}) that swaps exactly that subtree, validates the fragment parses, re-runs coherence + a fast inspect, and bumps a revision — with automatic escalation to section-level regeneration on failure (bolt's lesson: fragile patches need a full-rewrite fallback). This converts facts from observations into action targets ('overflow at hero.cta, snippet #412' → patch that node), eliminates whole-page regeneration for one-element fixes, and later powers a click-to-edit UI for free. Recs 2, 3, 5, 8, 9, 10 all build on it.
2. **Pipe runtime errors and failed requests into ViewportReport, with delta semantics** (small effort, high impact, phase: now)
   The #1 convergent feature across all four reports, and near-free given the existing headless pass. Extend the lib/inspect.js probe to capture window.onerror/unhandledrejection and console.error into errors:[{type,msg,src,line}], and failed resource loads (404 on /vault/* hrefs, broken fonts/images) into failedRequests:[{url,status}] — consider switching the lab to CDP (chrome-remote-interface) for source positions and to enable rec 9 later. Add Replit's delta semantics: tag each inspect of a slug with a monotonic revision id and return errorsSinceLastInspect separately, so the agent attributes new errors to its last change instead of re-investigating stale ones (the stale-telemetry weakness called out in Lovable, and the root of bolt's white-screen blind spot). A composed page that throws or 404s its CSS is currently invisible to the facts — this closes the loop's biggest hole.
3. **Save-gate inside book_save_page: deterministic validation + autofix + structured rejection** (medium effort, high impact, phase: now)
   Lovable's adherence-scan-with-auto-retry plus v0's <250ms deterministic autofixers, implemented as a gate in the MCP write tool so the model can't talk past it. On book_save_page (and book_patch): (a) validate every class/snippet reference against the vault index and every palette/font-pair token against parsed palettes, auto-rewriting near-misses via the existing buildSearchIndex — v0's lucide-embedding trick using the search index instead of a vector DB; (b) lint model-produced CSS for raw hex/rgb where a palette token exists and values off the spacing/type scale; (c) apply rule-based remedies for recurring inspect findings (smallTaps → pad from theme scale, tinyText → nearest type-scale step, img/pre/table hOverflow → max-width:100%/overflow-x:auto) then re-inspect; (d) only surviving violations return as a structured rejection the agent must fix before the write lands. Track v0's single KPI in book.json: % of briefs whose saved page has zero violations across iphone-15/ipad/desktop + coherence above threshold, logged per engine, so every prompt/tool change is measured. This mechanism is the bulk of v0's 93.87%-vs-64.71% reliability gap over raw Claude.
4. **Checkpoints as {seed, tokens, brief, snippet ids, fact summary} + rollback/roll-forward MCP tools** (small effort, high impact, phase: now)
   Out-Replit Replit at near-zero cost: their atomic checkpoint needs a git commit + Neon DB branch + serialized agent memory; deterministic compose means ours is a few hundred bytes that perfectly reproduce a page. On every agent write, lib/book.js already archives revisions/<n>.html — additionally store in the revision's manifest: the compose inputs (seed, theme tokens, snippet ids), the brief id/prompt as the 'commit message', and the post-save ViewportReport summary (violation counts per viewport). Expose book_restore_revision(slug, n) and book_bookmark(slug, label) as MCP tools plus one-click UI restore. Storing facts per revision makes versions comparable by facts ('rev 7: 0 overflows, rev 9: 3') — something no competitor can do — and gives a scrubable 'what the agent saw' history (Replit's video-replay equivalent, as text). Encode the doctrine from Lovable's and bolt's docs directly in tool descriptions: when an edit made facts worse, reverting is preferred over regenerating.
5. **Doom-loop guard in the MCP layer: fact-fingerprint loop detection with attempt memory** (small effort, high impact, phase: now)
   The signature failure of the whole category (bolt: 7–20M tokens on one bug; Lovable: $15–70 fix cascades; Replit: loops despite prompt-level reflection). Design Book can do it deterministically because facts are structured, which nobody with prose logs can: fingerprint each violation (kind + ref + viewport, e.g. 'hOverflow@hero.412@iphone-15'), record each fix attempt's fingerprint set on the brief in lib/book.js, and diff across attempts. If violations didn't decrease, the next tool response MUST include the prior attempts and why they failed (the documented anti-loop technique). If the same fingerprint survives N=3 attempts, the MCP server refuses further same-shaped writes and returns a directive: 'rollback to rev X (rec 4) and change strategy — different snippet family or layout primitive'. Harness-enforced, not prompt-suggested (Replit's lesson), and shared by both engines automatically via the tool surface. Small to build once recs 1–4 exist.
6. **Pick-from-3 deterministic variants at brief intake, and expose book_variants to the engines** (small effort, high impact, phase: now)
   v0's classic 3-variant flow front-loaded taste decisions but cost them 3x generation, so they de-emphasized it; /api/variants with seeds makes it literally free for Design Book. Wire the brief lifecycle: brief queued → server instantly composes 3 variant drafts (different seeds, optionally different palettes) → user picks in the UI (the Compose tab's 3-seed iframe grid already exists; connect it to briefs) → only then does the engine spend tokens, with the chosen variant as base and its theme tokens as constraints recorded on the brief. Also add book_variants as an MCP tool so the agent itself can propose-then-refine at zero generation cost. Surface the economics in the UI ('0 tokens, deterministic') — Replit's pricing-rage coverage shows users reward visible cheap paths; this makes the moat tangible.
7. **Vault registry: machine-readable catalog + cache-stable prompt prefix + hydration retrieval** (medium effort, high impact, phase: next)
   The mechanism behind v0's and Lovable's 'looks designed, on-brand' quality (shadcn Registry / design-system.json): constrain the model's vocabulary with a machine-readable system. Generate registry.json from the vault: per-snippet entries (role, variants, required tokens, one-line description, one usage example, composability constraints), the 55 palettes, the taste-token scale, and stack rules. Serve it three ways: (a) a compact summary as a constant, stable-ordered system-prompt prefix in engines/sdk.js to maximize Anthropic prompt-cache hits (v0 explicitly engineers for this); (b) a designbook-mcp resource + query_catalog tool for on-demand lookup — the hydration pattern: cheap deterministic retrieval feeds the model only the 3–5 relevant snippet specs, never raw files; (c) a curated exemplars/ set of 10–20 LLM-optimized snippet samples searchable via the existing /api/search (v0's 'code samples designed for LLM consumption'). Pair with a small human-curated book.md per book (taste defaults, banned snippet families, brand rules) read at session start — sparse and human-edited; Replit documents auto-generated context bloat as actively counterproductive.
8. **Deterministic design-lint facts: contrast, spacing-scale, overlap, alignment, dead controls** (medium effort, high impact, phase: next)
   The category's universal blind spot — every commercial loop verifies 'does it work', none verifies 'is it well-designed'; design-QA linting lives only in Figma plugins. Extend the lib/inspect.js probe with: (a) WCAG contrast of computed text color vs effective background per text node (flag <4.5:1); (b) spacing-scale adherence — computed margins/paddings off the theme's scale; (c) sibling/landmark bounding-box intersection — overlap detection, the main thing facts currently miss that screenshots would catch; (d) left-edge alignment clustering (x deltas of 1–7px within a container = misalignment); (e) the dead-control census, Replit's 'Potemkin interface' class via pure static analysis: buttons/links with no handler and no href, href="#", forms without action/submit, inputs outside forms, aria-controls pointing at missing ids. Fold all of it into the coherence score and the rec-3 save-gate. Zero tokens, pure deterministic-core advantage, and a marketing-visible differentiator no competitor has. Also enables a fact-gated 'done': a brief can't complete while design-lint violations remain (Lovable's publish-gate pattern, applied to layout/design instead of security).
9. **Persistent page sessions + book_interact for stateful verification** (medium effort, high impact, phase: next)
   Replit's single biggest verification win, and the one read-side capability the lab lacks: state. Keep the headless Chrome page alive across MCP calls (CDP session keyed by slug — the rec-2 CDP migration enables this) instead of cold-rendering per inspect. Add book_interact({slug, script}) executing a short Playwright/CDP script (click, fill, scroll, hover, open modal) against the live session and returning the standard structured facts + delta console/network errors after the interaction. This makes hover states, modals, accordions, mobile-nav toggles, and sticky-header geometry verifiable — interactive 'Potemkin interfaces' for design work — and Replit's numbers show persistence is what makes multi-step verification cheap (state from step 3 reused in step 12 without context tokens). Adopt their scheduling insight in the same change: cheap static facts on every draft; full perf mode (forced-reflow, long tasks, CLS) only at checkpoints/brief completion, encoded as policy in the MCP server so both engines inherit it.
10. **Zero-token Design Mode: per-element overrides on the taste-token scale, persisted as patch ops** (medium effort, high impact, phase: next)
   Lovable's client-side AST edits and v0's Design Mode are the most-loved cheap features in the category; Design Book's version is simpler because composition is deterministic and refs exist (rec 1). In ui/app.js, let the user click any element in the preview iframe (resolved to its data-db-ref) and edit: text content, palette-slot reassignment, radius/spacing/shadow steps from the taste layer's vocabulary, and per-section variant-seed re-roll. Apply live client-side (extending the instant re-theme path), persist as an overrides block keyed by ref in the page manifest, applied deterministically on render, committed as a normal revision (diffable/revertable per rec 4 — v0's rule). Hard rule, stated in the UI: anything expressible as a token/seed/text change never invokes a model. For NL structural asks from within Design Mode, auto-attach the selected element's outerHTML + its inspect-lab geometry facts to the brief — the superior facts-native analog of v0 attaching a cropped element screenshot.
11. **Context discipline bundle: verify-with-summarized-verdict, plan mode, parallel-read guidance** (small effort, medium impact, phase: next)
   Three small adoptions of the pattern that took Replit from 20 to 200+ minutes of useful autonomy. (a) Verifier isolation: a verify({slug, assertions}) MCP tool — assertions like 'no overflow at 393px, tap targets ≥44, coherence ≥ threshold, hero CTA above fold' — that runs inspect + design-lint + (later) interact internally and returns only a compact verdict {pass, topIssues:[{ref,kind,viewport,suggestedFix}]}; full multi-KB ViewportReports never enter the main trajectory. Plain code, no subagent model needed. In engines/sdk.js, instruct the agent to verify via this tool rather than raw book_inspect during iteration. (b) Plan/read-only mode: a brief flag where write tools (book_save_page, book_patch) are disabled but all read/fact tools work — Lovable's Chat-mode and bolt's Plan-mode pattern; near-free here since facts are deterministic, and perfect for 'why does this page feel off?' critiques the user approves before any spend. (c) Encode Lovable's context rules in tool descriptions: batch independent reads in parallel, never re-fetch a page already in context, edit by ref scope not whole pages.
12. **Brief back-testing as a regression suite + JSONL trajectory recording** (medium effort, medium impact, phase: later)
   Lovable back-tests prompt changes against historical queries with manual verification; Replit replays recorded failure trajectories against harness fixes. Design Book automates both because the core is deterministic. (a) Keep the briefs.json history as a corpus; whenever compose logic, snippet sources, the registry, or system prompts change, re-run the corpus and assert on structured facts: zero overflows at 393/820/1440, tap targets ≥44px, no design-lint violations, coherence floor, KPI (rec 3) not regressed. Compose-path changes run entirely model-free; agent-path changes need one cheap pass. (b) Log every MCP call + args + returned facts per session as plain JSONL beside the book; when the fact extractor or compose logic changes, replay recorded sessions to see which past agent decisions would have changed — exact reproduction, where Replit has to fake it with snapshots. This is the meta-loop that let both vendors iterate their harnesses safely, and it future-proofs every rec above.
